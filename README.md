## 1. 기존 금융권의 지급결제 시스템
'지급결제'란 경제주체들이 경제활동에 따른 채권채무관계를 지급수단을 이용하여 해소하는 행위를 의미합니다. 현금 거래와 달리 신용카드나 계좌이체는 정보와 자금의 흐름이 분리된 복합적인 과정을 거칩니다.

### (1) 지급결제 개념
* **지급 (Payment):** 경제주체가 대금 지불을 위해 카드, 계좌이체 등 지급수단으로 '지급지시 전문'을 전송하는 행위
* **청산 (Clearing):** 금융기관 간 채권·채무를 계산하는 과정. 여러 거래내역을 상계(Netting)하여 각 은행이 최종적으로 주고받을 금액을 확정
* **결제 (Settlement):** 확정된 차액(Net 금액)을 중앙은행 지준계좌를 통해 실제로 이체하여 채권·채무를 종결시키는 최종 단계

### (2) 시스템별 프로세스 예시
1.  **소액결제시스템 (DNS, 이연 차액결제)**
    * **지급:** 채무자 은행 → 전자금융공동망(결제원) → 채권자 은행
    * **청산:** 당일 야간, 결제원에서 각 은행별 거래 상계하여 차액(Position) 산출
    * **결제:** 익영업일 한국은행 지준계좌에서 차액 결제 (은행 간 정산 완료)
2.  **거액결제시스템 (RTGS, 실시간 지준결제)**
    * 채무자 은행 → 한국은행(BOK-Wire+) → 지준계좌 간 즉시 이체
3.  **신용카드**
    * 가맹점 → VAN사 → 카드사 → 카드사 은행 → 전자금융 공동망 → 가맹점 은행

## 2. 비트코인의 지급결제 시스템
비트코인은 **지급/청산/결제가 통합**되어 거의 동시에 일어납니다. 전통 금융은 자산과 메시지를 분리하여 '지급지시 전문'을 전송하지만, 비트코인은 UTXO 기반으로 메시지와 가치를 통합하여 전송하기 때문입니다.

* **지급:** UTXO에 개인키로 서명하여 네트워크에 전파함으로써 디지털 자산 가치 자체를 전송
* **청산:** 중앙기관의 청산 대신 네트워크의 각 노드들이 프로토콜에 따라 트랜잭션의 유효성을 실시간 검증
* **결제:** 채굴자들이 작업증명(PoW)을 통해 거래를 블록에 생성(On-chain). 이후 컨펌이 쌓일수록 확률적 최종성 강화됨

## 3. 기존 금융과 블록체인 레이어 비교
모든 거래를 메인넷에 기록할 때 발생하는 비용과 속도 문제를 해결하기 위해, 블록체인은 전통 금융과 유사한 **계층적 구조**를 채택하고 있습니다.

### (1) RTGS (실시간 지준이체) ↔ Layer 1 (On-chain)
> **"모든 거래가 즉시, 개별적으로, 최종 확정된다."**
* **RTGS:** 한은금융망 등을 통해 은행 간 거액 결제를 건건이 즉시 처리
* **L1:** 트랜잭션이 블록에 포함되는 순간 그 자체로 결제 완결성을 갖춤

### (2) DNS (익일 차액결제) ↔ Layer 2 (Rollup/Off-chain)
> **"작은 거래는 밖에서 모으고(Batch), 결과만 상위 계층에 보고한다."**
* **DNS:** 금융결제원이 수만 건의 거래를 상계하여 당일 'A은행이 B은행에 줄 순액(net)'만 산출한 뒤, 익일 한국은행에 보고하여 지준계좌 간 처리
* **L2 (Rollup):** 수천 건의 트랜잭션을 오프체인에서 실행하고, 그 결과값들을 하나로 묶어(Batch) L1에 기록

### ⚠️ 계층 구조의 리스크와 발전 방향
계층 구조는 효율적이나 중간 운영 주체의 신뢰도에 의존해야 한다는 리스크가 존재합니다. (예: 빗썸 오지급 사태)
이를 극복하기 위해 블록체인 생태계는 다음과 같은 기술로 **'인간의 개입이 불가능한 시스템'**으로 발전 중입니다.
* **기술적 대안:** 영지식증명(ZKP), 탈중앙 시퀀서, 자산증명(PoR), 계정추상화(AA) 등
